## 2. 产品定义

V1 是一个保留现有 QQ 三国视觉素材和横版操作表现、逐步重写逻辑层的本地单机怀旧版。它不是官服复刻，也不是联网游戏。

玩家应能从新档开始，完成以下完整循环：

```text
创建新档
  → 接主线/支线任务
  → 移动、打怪、使用技能
  → 获得经验、金币、物品和装备
  → 装备、买卖、强化
  → 解锁地图
  → 挑战 Boss 和单人副本
  → 保存、退出、重新启动并继续
  → 达到 30 级并完成 V1 主线结局
```

目标首次通关时长：主线约 4～6 小时；完成全部支线和两次副本首通约 6～8 小时达到 30 级。任何主线门槛不得要求超过 15 分钟纯刷怪。

### 2.1 平台与最低性能基线

- 发布目标为 Windows 10/11 64 位，Godot 3.5.3 GLES3，离线键鼠游玩。
- 最低配置口径：双核 2.0 GHz x64 CPU、4 GiB RAM、支持 OpenGL 3.3 的 Intel HD 4000 级或同等 GPU、800×600 分辨率、普通机械硬盘。
- Phase 0 必须记录性能机的 CPU/GPU/RAM、系统版本、驱动、显示模式和构建哈希；Phase 39 的 NFR-02 结论必须来自这台最低配置或经记录证明不高于它的机器，开发机结果不能替代最低配置结论。
- 若最低配置实测不能达到 NFR-02，必须先调整表现预算或正式修改本节，不能只降低测试阈值。

## 4. 核心用户需求

### FR-01 启动与恢复

- 从干净安装目录启动时进入新游戏/继续游戏界面。
- 资源、内容或存档异常必须给出可读错误码和恢复动作，不能停在黑屏或无限等待。
- 加载失败不得部分修改当前运行状态。

### FR-02 常驻游戏壳

- 地图切换期间 `GameRoot`、玩家、UI 和相机各只有一个实例。
- 地图只替换 `WorldRoot` 下的世界子场景。
- 任何地图不得再自带第二份玩家或常驻 UI。

### FR-03 地图与传送

- 所有地图、入口和出生点由稳定 ID 引用。
- 重复碰撞、同帧双传送、传送中死亡/暂停/自动战斗等情况必须在有界时间内成功或回滚。
- 传送失败保留原地图，解除黑屏和输入锁，并显示原因。

### FR-04 玩家移动与复活

- 保留左右移动、跳跃、爬梯、下平台和现有主要动画表现。
- 玩家死亡后按地图/检查点定义复活，不固定回 `Level1`。
- 手动输入和自动战斗共同使用 `PlayerIntent` 接口，手动输入可立即停止自动控制。

### FR-05 属性与成长

- 等级、经验、HP、MP 和派生属性只有一个权威来源。
- 使用显式 1～30 级经验表；支持一次获得大量经验跨多级；满级不出现 31 级。
- 主线结束时目标等级为 27～29，完成全部支线和两次副本首通可稳定达到 30。

### FR-06 战斗

- 玩家、普通敌人和 Boss 的伤害都经过同一 `CombatAction → Damage/Effect → Vitals` 管线。
- 物理攻击、防御、技能倍率、暴击、状态效果和死亡结果可由纯数据测试复现。
- 每个命中窗口、击杀和奖励只结算一次。

### FR-07 技能

- 8 个技能均有独立 ID、等级门槛、MP 消耗、冷却、范围、倍率、命中时点、动画和音频映射。
- V1 的 8 个技能解锁等级固定为 1、3、6、10、14、18、23、28；偏离必须先修改本 Master Spec。
- MP 不足、冷却中、目标无效、施法被取消时返回明确结果，不产生半次攻击。

### FR-08 敌人和生成

- 10 个普通敌人必须至少在数值、掉落、攻击提示或行为职责上形成独立遭遇价值。
- AI 支持发现、追击、攻击、目标失效、脱战、死亡和重生。
- 生成器负责生成和清理，不负责第二次发放经验或掉落。

### FR-09 物品与背包

- 堆叠物使用 `item_id + quantity`；装备使用独立 `instance_id + item_id + enhance_level`。
- 背包容量、堆叠、拆分、消耗、拾取和满包行为必须保持数量守恒。
- 任务物品不可卖、不可丢；满包时任务进度不能静默丢失。

### FR-10 装备与强化

- 10 个装备槽位必须由定义表校验职业、等级和槽位。
- 穿上、替换、卸下和重复加载都从基础值重新计算派生属性，禁止用反复加减造成漂移。
- 强化仅放大装备定义中的主属性；倍率以 basis points 显式表固定为 10000、10400…14000，非负整数属性统一四舍五入，成本不得散落为代码公式。
- 每个槽位的显式 `power_score` 必须满足 `下一档 +2 ≤ 上一档 +10 ≤ 下一档 +3`，避免旧装备立即作废或永久最优。
- 同档装备从 +0 到 +10 的总金币必须在基础买价的 1.90～2.10 倍之间，并消耗 8～12 个同档强化材料。
- 标准全清路线在扣除主线必需消费后，保证可把 1～2 件终档装备升到 +10；全身 +10 需要可选重复刷取。

### FR-11 奖励和掉落

- 敌人、任务、Boss、副本、商店都通过同一个原子事务服务修改金币和物品。
- 主线必需任务物品在任务激活时必须保证掉落，不使用低概率卡主线。
- 关键推进装备由任务、商店或首通奖励保证；随机掉落只提供额外收益。

### FR-12 世界状态、NPC 与对话

- 世界标记、地图解锁、一次性奖励、Boss 击败和副本首通进入存档。
- NPC 通过稳定 ID 注册；同一时刻只允许一个交互锁。
- 对话可根据任务和世界标记选择分支，但 V1 不做剧情选择和多结局。

### FR-13 任务

- 任务状态固定为 `locked → available → active → completable → completed`。
- V1 目标类型为对话、击杀、收集、进入地图、击败 Boss、完成副本。
- 前置任务图无环、所有任务从新档可达、奖励提交幂等。

### FR-14 商店与经济

- 买入、卖出、强化先校验金币、材料、容量和物品权限，再全成或全败。
- 装备卖价固定为 `max(1, floor(base_buy_price × 0.25))`；卖出不返还强化投入。
- 不允许通过买卖循环获得正收益。

### FR-15 Boss 与副本

- Boss 支持阶段、技能预警、失败重置、玩家离场、死亡和一次性奖励。
- 副本支持进入、波次、检查点、失败、退出、重试、首通和重复奖励。
- V1 不在战斗中间保存全部瞬时敌人；退出后从最近检查点或入口恢复。

### FR-16 自动战斗

- 状态包括关闭、找目标、接近、施法、恢复、拾取和脱困。
- 仅在地图配置的自动战斗区域内工作，不跨传送点追怪。
- 背包满、死亡、无药、任务完成、无可达目标、离开活动半径或手动关闭时自动停止。
- 自动战斗使用与手动相同的 MP、冷却、药品、伤害、掉落和奖励规则；目标产出为手动基准的 70%～90%。

### FR-17 保存与加载

- 存档只包含 JSON 基础类型和稳定 ID，不保存 NodePath、场景对象、贴图或显示名身份。
- 写入流程为规范化、选择非活动世代、写临时文件、刷新并关闭、回读校验、替换非活动世代；当前有效世代在新世代完整落盘前始终保留。
- V1→V2 迁移保留原文件备份并生成迁移报告；未知 ID 不得静默丢弃。
- 同一快照应用 1、2、10 次的规范状态、属性、节点、信号和计时器数量必须一致。

### FR-18 设置与音频

- 音量、显示和按键设置写入独立 `ConfigFile`，不随游戏存档回滚。
- `AudioManager` 只负责跨地图音乐、全局 UI 音效和必要的播放器池；角色局部音效由角色场景负责。

## 5. 非功能要求

### NFR-01 确定性与可诊断性

- 自动测试使用固定随机种子和固定物理帧。
- 每个传送、保存、奖励和遭遇都有唯一操作 ID 与一次终态结果。
- 失败日志包含错误码、操作 ID、地图/内容 ID 和最近事件，不依赖截图猜原因。

### NFR-02 性能

- 声明的最低配置上，正常战斗中位帧率至少 60 FPS，1% low 至少 45 FPS。
- 除加载阶段外，单帧不超过 250 ms。
- 保存 p95 不超过 100 ms、最大不超过 250 ms；加载并应用 p95 不超过 3 秒。
- 传送必须在 300 帧或 5 秒内结束为成功或失败。

### NFR-03 稳定性

- 2 小时自动浸泡中至少完成 1,000 次传送、100 次跨进程存档恢复和 10,000 场普通战斗。
- 热身后活动节点、Timer 和信号连接净增长为 0。
- 进程内存增长同时满足不超过 5% 且不超过 32 MiB。

### NFR-04 日志质量

- 发布候选中 `ERROR:`、`SCRIPT ERROR:`、Parser Error、missing resource 为 0。
- Warning 只能通过带负责人、原因和到期 Phase 的白名单；到期未清零即阻断。
- 退出码为 0 但没有 `TEST_RESULT` 终结记录，或出现未分类错误，均视为失败。

### NFR-05 离线与安全边界

- 游戏运行时不联网、不下载脚本、不加载存档指定的任意资源路径。
- 存档中的 ID 必须先通过 `ContentRegistry` 映射到受信任的 `res://` 资源。
- 只用 JSON 解析进度数据；不反序列化可执行对象。
- 写操作限于 `user://`；开发工具和导出模板固定版本并记录哈希。

### NFR-06 兼容与可回滚

- 每个 Phase 保持游戏可启动，并最多引入一次有界迁移。
- 结构改动与行为改动分开提交；Gate 前保留可回滚点。
- 新代码只能依赖六个目标 Autoload；旧 Autoload 通过例外清单只减不增。

## 8. 每个 Phase 的完成定义

Phase 只有同时满足以下条件才算完成：

1. Phase Spec 锁定可观察行为、范围、失败体验和量化验收。
2. Phase Design 锁定边界、数据、接口、状态机、迁移和回滚。
3. Implementation Plan 使用测试先行的小步骤和精确命令。
4. 新测试先失败再通过；相关旧测试仍通过。
5. 退出码、终结 JSON 和日志分类三项同时为绿。
6. 涉及内容时，ID、schema、引用、计数和图连通校验通过。
7. 涉及状态时，重复应用和存档往返测试通过。
8. 没有把业务规则放回 UI、地图脚本或新的全局单例。
9. 只提交本 Phase 的有意文件，不混入编辑器生成或用户未确认的资源差异。
10. 生成 Acceptance Report，记录命令、结果、已知警告和回滚点。

## 9. 当前基线与差距
