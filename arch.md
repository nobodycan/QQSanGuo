# 结论先说

**有搞头，但要把目标定义成“QQ三国怀旧单机版/同人原型”，不要定义成“追平当前官服的单机复刻”。**

对你这种“自己电脑上优化后玩、不联网、不商用”的需求，我给它的可行性评分是：

| 目标 | 可行性 |
|---|---:|
| 把现有工程跑起来，体验地图、打怪、技能 | **8/10** |
| 修成一个有刷怪、掉落、装备、升级、存档闭环的单机版 | **7/10** |
| 增加几个职业、地图、副本、元神等精选系统 | **5/10** |
| 还原现在官服大部分内容 | **2/10** |
| 做成可长期运营的联网版 | **1/10** |

最重要的判断是：

> 这个项目不是“完成度较低的完整 QQ三国”，而是一个已经做出视觉表现和基本战斗的 **Godot 3 原型工程**。  
> 它最大的价值是帮你省下了“角色动画、部分地图、UI、基础操作和素材整理”的起步工作，而不是提供现成的游戏系统框架。

项目地址：[Time1996/QQSanGuo](https://github.com/Time1996/QQSanGuo)

- `D`
- `F`
- `W`
- `E`

但这里有一个很关键的问题：

技能伤害变量虽然存在，但没有完整接入伤害结算。攻击函数里根据键位切换的主要是：

- 动画

也就是说，目前不同技能在实战里很可能主要区别是动画表现，而不是完整的：

- 技能倍率
- 蓝量消耗
- 冷却时间
- 攻击范围
- 多段判定
- 异常状态
- 击退
- 霸体
- 持续伤害
- 技能等级

## 5. 技能数据层的问题

项目有：

- `Data/skill.json`
- `Data/SkillsFactory.gd`
- `system/skill/SkillsProperty.gd`

这是正确方向：试图把技能数据从角色脚本中抽离出来。

但是 `skill.json` 目前只有少量条目：

- 横击剑
- 饮血剑舞
- 回风拂柳

而且 `SkillsFactory.gd` 中有明显的不一致：

```gdscript
temp[SkillsPropertyList.Name] = data['skill_ame']
```

JSON 里没有 `skill_ame` 字段，技能名称实际上是 JSON 对象的键。这部分如果真正调用，很可能直接报错。

另外，大量技能逻辑依然硬编码在 `Steve.gd` 中，数据工厂和实际战斗没有完全连接起来。

### 这意味着什么

你后续如果要加官服新技能，不能简单地：

1. 添加一条 JSON；
2. 放入技能图片；
3. 就能自动使用。

你需要先把现有技能系统重构成：

```text
技能数据
  ↓
技能实例
  ↓
施法条件
  ↓
扣除蓝量/进入冷却
  ↓
攻击判定
  ↓
伤害公式
  ↓
状态效果
  ↓
动画与音效
```

这是整个项目最值得优先重做的模块之一。

# 七、存档系统风险比较大

存档脚本：

[SaveState.gd](https://github.com/Time1996/QQSanGuo/blob/main/SaveState.gd)

它使用 Godot 3 的 `File` 类，把 `Persist` 组中的节点逐行存成 JSON。

优点：

- 已经有本地存档思路
- 可以保存玩家位置和部分属性
- 可以保存部分物品节点
- 不依赖数据库

问题：

1. 存档依赖节点路径和场景结构。
2. 读取时会先切换场景，然后固定等待 1 秒。
3. 大量逻辑依赖根节点名称。
4. 存档格式没有版本号。
5. 缺少存档损坏保护。
6. 缺少备份存档。
7. 没有完整保存技能、任务、装备、配置等数据。
8. 场景加载稍慢或电脑环境变化，固定等待 1 秒可能不可靠。
9. 存档数据和场景节点耦合，重构地图后旧存档容易失效。

### 建议

不要继续扩大当前 Persist 节点方案。应当改成一个统一的 `GameState`：

```text
SaveData
├── version
├── player
│   ├── level
│   ├── exp
│   ├── hp/mp
│   ├── attributes
│   └── position/map
├── inventory
├── equipment
├── skills
├── quests
├── world_flags
└── settings
```

保存时一次性序列化，读取时根据版本迁移。

对于你这种单机目标，JSON 就足够了，暂时没必要引入数据库。

# 九、Godot 版本问题

项目的 `project.godot` 使用：

```text
config_version=4
KinematicBody2D
File.new()
yield(...)
export var
onready var
PoolStringArray
```

这明确属于 **Godot 3.x 工程**，不是 Godot 4 工程。

## 建议先用 Godot 3.5.3

官方下载：

[Godot 3.5.3 下载页](https://godotengine.org/download/archive/3.5.3-stable/)

不要一开始就直接拿 Godot 4.6 打开并转换。

原因是升级到 Godot 4 后要处理：

| Godot 3 | Godot 4 |
|---|---|
| `KinematicBody2D` | `CharacterBody2D` |
| `File` | `FileAccess` |
| `yield` | `await` |
| `export var` | `@export var` |
| `onready var` | `@onready var` |
| `instance()` | `instantiate()` |
| `change_scene()` | `change_scene_to_file()` |
| Pool 数组 | Packed 数组 |
| AnimationTree/节点属性 | 部分接口变化 |

Godot 官方也明确说明，3 到 4 的迁移不能保证所有脚本和场景自动正确转换：[Godot 3 升级到 Godot 4](https://docs.godotengine.org/en/stable/tutorials/migrating/upgrading_to_godot_4.html)

### 额外依赖风险

`project.godot` 中启用了：

```text
codeandweb.texturepacker
```

但仓库没有看到完整的 `addons` 目录。

这可能导致编辑器提示缺少插件。通常不会阻止已经生成好的纹理资源运行，但可能影响重新编辑或导入 TexturePacker 图集。

如果打开工程时报插件错误，可以先：

1. 备份项目；
2. 在 Godot 3.5.3 中导入；
3. 禁用缺失插件；
4. 等项目正常运行后再处理图集工作流。

- 副本战力与贡献判定
- 元神
- 奥义
- 八阵图
- 商行
- 交易冻结
- 职业平衡
- 多种日常活动

官方公告：[2026 年 4 月 14 日体验服更新](https://sg.qq.com/webplat/info/news_version3/159/24483/24484/24485/m17776/202604/984552.shtml)

相比之下，GitHub 项目目前主要具备：

- 单一剑侍角色
- 少量技能
- 基础怪物
- 一个 Boss
- 少量地图
- 基础背包装备
- 初步存档

因此不能把“增加官服新特性”理解成给现有代码加几个模块。很多现代功能依赖几十个底层系统。

# 十一、哪些新特性值得你单机化

你的优势是：**不需要照搬网游逻辑，可以把复杂系统改造成单机版本。**

## 第一档：非常值得做

### 1. 自动打怪

现有项目已经有：

- Tab 选怪
- 自动接近目标
- 怪物追击
- 技能键位
- 拾取范围

因此自动打怪只需补：

- 搜索最近怪物
- 选择技能优先级
- 检查蓝量和冷却
- 自动拾取
- 血量低时吃药
- 找不到怪物时巡逻

这是比较适合当前基础的功能。

难度：**低到中**

---

### 2. 完整任务系统

单机版最需要：

- 主线任务
- 支线任务
- 击杀任务
- 收集任务
- NPC 对话
- 奖励经验和装备
- 解锁地图

难度：**中**

但价值非常高。任务系统完成后，这个项目才真正从战斗 Demo 变成游戏。

---

### 3. 完整掉落表

现在怪物基本使用通用掉落场景。

建议改成：

```text
怪物ID
├── 掉落物ID
├── 掉落概率
├── 最小数量
├── 最大数量
└── 是否必掉
```

然后给不同地图的怪物配置：

- 药品
- 装备
- 材料
- 三国币
- 任务物品

难度：**低到中**

---

### 4. 装备强化简化版

不需要完全复制官服，可以做：

- `+0 ~ +10`
- 强化增加基础攻击/防御
- 消耗材料和三国币
- 高等级有失败概率
- 单机模式允许失败不降级

难度：**中**

---

### 5. 单机副本

把官方多人副本改成：

- 单人进入
- 几波小怪
- 机关
- 最终 Boss
- 首通奖励
- 每日次数可取消

难度：**中**

现有地图、怪物、Boss 已经可以提供基础。

---

## 第二档：可以做，但要先重构

### 1. 多职业

当前明显以剑侍为核心：

- 物品的职业字段大量是 `js`
- 技能动画和脚本围绕剑侍
- 伤害逻辑偏物理职业
- 玩家场景只有一套主要实现

要增加：

- 豪杰
- 阴阳士
- 仙术士
- 游侠
- 唤灵等新职业

必须先把玩家角色拆成：

```text
PlayerController
CombatController
CharacterStats
SkillController
EquipmentController
AnimationAdapter
JobDefinition
```

否则每增加一个职业，就要复制整个 `Steve.gd`，后面几乎没法维护。

难度：**中到高**

建议第一阶段只保留剑侍，最多再加一个远程职业验证架构。

---

### 2. 元神/宠物简化版

可以做成：

- 元神跟随
- 自动攻击
- 元神等级
- 3~5 个技能
- 给玩家加属性
- 出战/休息

不要一开始做：

- 元神融合
- 多代成长计算
- 复杂阶级
- 技能继承
- 市场交易
- 随机洗练

难度：**中到高**

---

### 3. 家园

单机版可以简化为：

- 独立家园地图
- 放置家具
- 种植
- 定时收获
- 仓库

但是家具编辑器和存档结构会比较耗时间。

难度：**中到高**

---

## 第三档：不建议做

如果只是自己玩，以下官服系统性价比极低：

- 跨服
- 军团战
- 商行
- 寄卖行
- 交易冻结
- 排行榜
- 组队匹配
- 邮件
- 玩家交易
- 防盗防骗
- 充值活动
- 多人围猎
- 网络同步
- 反作弊

单机版可以直接用 NPC 商店和固定掉落代替经济系统。

# 十二、建议的实际改造路线

## 阶段 0：先验证工程能否正常运行

建议：

1. Fork 到你自己的 GitHub。
2. 下载 Godot 3.5.3。
3. 克隆仓库。
4. 用 Godot 3.5.3 导入 `project.godot`。
5. 等待全部素材重新导入。
6. 记录所有报错。
7. 禁用缺失的 TexturePacker 插件。
8. 运行默认场景 `Scene/bajun.tscn`。
9. 测试移动、技能、打怪、掉落、背包、传送、存档。
10. 给原始可运行状态打一个 `baseline` Tag。

预计：**半天到三天**，取决于缺失资源和导入错误。

---

## 阶段 1：不要加功能，先把工程稳定下来

优先修：

- 空节点引用
- 地图名称硬编码
- 背包 UI 查找
- 玩家等级不同步
- 技能 JSON 字段错误
- 存档读取固定等待 1 秒
- 死亡后固定回 `Level1`
- 角色蓝量初始化
- 怪物脱战
- 技能伤害未接入
- 缺失的传送口
- UI 在不同场景重复创建

目标是：

> 连续玩 30 分钟，不报错、不丢存档、不因切地图丢 UI。

预计：**1～3 周业余时间**。

---

## 阶段 2：重构核心框架

建议先拆出以下模块：

```text
autoload/
├── GameState.gd
├── SaveManager.gd
├── SceneManager.gd
├── InventoryManager.gd

└── AudioManager.gd

player/
├── PlayerController.gd
├── PlayerCombat.gd
├── PlayerStats.gd
├── PlayerSkills.gd
└── PlayerEquipment.gd

combat/
├── DamageCalculator.gd
├── Hitbox.gd

world/
├── MapDefinition.gd

```

不需要一次重写全部，而是逐步替换旧系统。

预计：**3～8 周业余时间**。

---

## 阶段 3：补齐单机闭环

优先顺序建议是：

1. 伤害公式
2. 技能冷却和蓝耗
3. 掉落表
4. 完整装备数据
5. 稳定存档
6. NPC 对话
7. 任务系统
8. 商店
9. 地图解锁
10. 副本
11. Boss 技能
12. 自动打怪

做到这里，项目就已经有“自己长期玩”的价值了。

## 阶段 4：再挑官服新系统

一次只选一个：

- 新职业
- 元神
- 强化
- 翅膀
- 自动打怪
- 单人副本
- 家园
- 新地图
- 新 Boss

不要同时开多个坑。

# 十三、要不要升级到 Godot 4

我的建议是：

## 短期：不要

先用 Godot 3.5.3 把项目跑通和修稳。

## 中期：视情况升级

满足以下条件再考虑：

- 你确认会长期维护
- 存档系统已经独立
- 技能系统已经数据化
- 玩家和 UI 解耦
- 地图切换稳定
- Git 已做好迁移前备份

Godot 4 的长期维护、性能分析和现代 API 更好，但这个项目的 Godot 3 写法很多，升级工作量不会小。

如果你熟悉 Godot，预计迁移约 **1～3 周**；如果第一次接触 Godot，可能需要更久。

# 十四、版权和使用边界

仓库显示 CC0，但 README 明确写着：

> 素材源自《QQ三国》，仅供个人学习研究使用。

[项目 README](https://github.com/Time1996/QQSanGuo/blob/main/README.md)

这里要区分：

- 作者自己写的代码可以按仓库许可证理解。
- 腾讯游戏的角色、地图、音乐、UI、技能特效等素材，不会因为被上传到 CC0 仓库就自动变成 CC0。

你自己在电脑上研究、修改和游玩，实际风险相对较低，但仍建议：

- 不公开发布包含原版资源的安装包
- 不收费
- 不开服
- 不宣传为官方版本
- 不上传新的官服完整资源包
- 不尝试绕过官方客户端保护或反作弊
- 自己的代码仓库最好保持私有
- 如果公开代码，尽量把原版资源排除

“非商用”不等于自动获得版权授权，只是你的使用场景相对克制。

# 十五、最终判断

## 值得接手的原因

- 已经有不错的 QQ三国视觉表现。
- 基础横版操作、技能、怪物和掉落能用。
- 有部分地图、NPC、UI 和装备素材。
- Godot 对个人开发友好。
- 你的目标是单机，不需要解决服务器和网络同步。
- 可以把复杂官服系统简化为适合单人的版本。

## 不值得接手的情况

如果你期望的是：

- 下载后直接玩
- 很快补齐官服内容
- 不想写大量代码
- 不想学习 Godot
- 不想整理数据和地图
- 希望现有架构直接扩展十几个大型系统

那这个项目不合适。

## 我的实际建议

**可以搞，但应该把它当成“带有大量 QQ三国素材和动画的半成品原型”，而不是半成品游戏。**

最合理的第一版目标是：

> 一个只支持剑侍、包含 5～10 张地图、20～30 个任务、10 种怪物、3～5 个 Boss、完整装备掉落、技能升级、自动打怪和稳定存档的单机怀旧版。

这个规模个人有机会做完，也确实能玩。

如果一开始目标就是：

> 五六个职业、元神、奥义、八阵图、家园、军团、跨服、商行、全部地图和最新资料片

大概率会在重构阶段就弃坑。

**综合评价：有 60% 的代码/素材参考价值，但只有约 15%～25% 的完整游戏系统完成度。最适合“保留表现层、逐步重写逻辑层”。**
